# OMS 技能库 - 业务场景配置指南

> 从营销和产品视角，讲述如何配置业务流程解决实际问题

---

## 一、平台集成

OMS 系统已支持与主流电商及营销平台的对接，实现多平台数据统一管理。

### 京东开放平台对接 (oms-jd-integration)

**功能覆盖**：
- 订单同步：POP订单、预售订单、货到付款订单、门店自提、跨境购
- 库存管理：可售库存同步、预占与释放
- 物流追踪：运单轨迹查询、物流状态更新
- 退货处理：退货申请、退款查询、退货状态同步

**接入方式**：OAuth 2.0 + MD5签名认证

### 小红书开放平台对接 (oms-xhs-integration)

**功能覆盖**：
- 笔记数据：曝光/互动数据查询，效果评估
- KOL合作：蒲公英平台合作订单管理，投放效果追踪（CPE/CPI）
- 薯店订单：电商订单同步，状态映射
- 小程序订单：微信小程序订单同步，UTM归因
- 归因计算：基于UTM参数，30天归因窗口，权重 0.15

**接入方式**：OAuth 2.0 + MD5签名认证

## 二、核心业务场景

### 场景1：新客户首单转化

**用户痛点**：新客户下单后，不知道其消费偏好，无法精准推荐，导致复购率低。

**解决思路**：

```
客户下单 → 自动识别身份 → 查询历史画像 → 匹配促销 → 路由最优仓库
```

**配置方案**：

| 步骤 | 技能 | 解决的问题 |
|------|------|-----------|
| 1 | 订单捕获 | 自动清洗多平台订单格式 |
| 2 | One-ID归一 | 识别是否是新人/老客 |
| 3 | 库存防超卖 | 确保热门商品有货 |
| 4 | 促销计算 | 自动匹配最优优惠 |
| 5 | 订单路由 | 选择最近仓库发货 |

**业务价值**：新客首单转化率提升 → 客单价提升

---

### 场景2：限时大促防超卖

**用户痛点**：大促期间库存不同步，多平台超卖引发客诉。

**解决思路**：

```
实时库存查询 → 预扣库存 → 订单成交 → 释放未支付库存
```

**配置方案**：

| 步骤 | 技能 | 解决的问题 |
|------|------|-----------|
| 1 | 库存实时查询 | 各渠道库存一目了然 |
| 2 | 超卖风险检测 | 库存不足时预警 |
| 3 | 预扣库存 | 锁定库存防止超卖 |
| 4 | 订单路由 | 就近发货减少时效 |

**业务价值**：超卖投诉降为0 → 库存周转率提升

---

### 场景3：跨渠道退货（Anywhere Return）

**用户痛点**：客户在A平台买的商品，只能在A平台退货，不方便导致差评。

**解决思路**：

```
退货申请 → 资格校验 → 就近门店退/上门取 → 质检分类 → 库存再流转
```

**配置方案**：

| 步骤 | 技能 | 解决的问题 |
|------|------|-----------|
| 1 | 退货资格检查 | 判断是否在退换货期内 |
| 2 | 退款处理 | 原路返回/账户余额 |
| 3 | 物流追踪 | 实时查看退货进度 |
| 4 | 质量检验 | 区分残次品等级 |
| 5 | 库存再上架 | 合格品自动上架销售 |

**业务价值**：退货便利度提升 → 好评率提升 → 残次品及时处理

---

### 场景4：会员积分与促销联动

**用户痛点**：促销和积分各自为政，无法根据会员等级差异化营销。

**解决思路**：

```
识别会员等级 → 积分抵扣 → 优惠券叠加 → 阶梯优惠
```

**配置方案**：

| 步骤 | 技能 | 解决的问题 |
|------|------|-----------|
| 1 | One-ID归一 | 识别会员身份和等级 |
| 2 | 促销引擎-积分计算 | 积分抵扣金额 |
| 3 | 促销引擎-优惠计算 | 叠加优惠券 |
| 4 | 促销引擎-发放优惠券 | 下单后自动发券 |

**业务价值**：会员活跃度提升 → 复购频次提升

---

### 场景5：O2O分润结算

**用户痛点**：线上订单线下门店履约，佣金分润规则复杂，容易出错。

**解决思路**：

```
订单成交 → 识别履约方 → 计算分润比例 → 生成结算单 → 财务凭证
```

**配置方案**：

| 步骤 | 技能 | 解决的问题 |
|------|------|-----------|
| 1 | 订单路由 | 识别最优履约方（门店/仓库） |
| 2 | 佣金计算 | 按渠道/品类/履约方计算佣金 |
| 3 | 分润分配 | 多方分润自动分配 |
| 4 | 结算单生成 | 生成对账单 |
| 5 | 凭证生成 | 对接财务系统 |

**业务价值**：分润结算效率提升 → 财务差错率降低

---

### 场景6：每日对账

**用户痛点**：多平台订单数据与系统不一致，财务对账耗时。

**解决思路**：

```
拉取平台账单 → 与系统订单比对 → 差异标红 → 生成凭证 → 导出报表
```

**配置方案**：

| 步骤 | 技能 | 解决的问题 |
|------|------|-----------|
| 1 | 账单核对 | 自动匹配差异订单 |
| 2 | 凭证生成 | 生成财务凭证 |
| 3 | 报表导出 | 多维度统计报表 |

**业务价值**：对账时间从天级降到分钟级

## 三、预置业务流程（开箱即用）

### 流程A：订单履约全流程

**适用场景**：客户下单到收到货的完整过程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  订单捕获   │ →  │  用户识别   │ →  │  库存检查   │
│ (多平台订单)│    │ (One-ID)    │    │ (防超卖)    │
└─────────────┘    └─────────────┘    └─────────────┘
        │                                    ↓
        │                              ┌─────────────┐
        │                              │  促销计算   │
        │                              │ (优惠券)    │
        │                              └─────────────┘
        │                                    ↓
        │                              ┌─────────────┐
        │                              │  订单路由   │
        │                              │ (最优仓库)  │
        │                              └─────────────┘
        │                                    ↓
        │                              ┌─────────────┐
        └───────────── 成交后 ───────→│  佣金计算   │
                                      │ (分润)      │
                                      └─────────────┘
```

**解决的问题**：
- 多平台订单格式混乱 → 统一标准化
- 不知道客户是谁 → 身份识别
- 热门商品超卖 → 库存预扣
- 优惠叠加混乱 → 规则引擎
- 发货路径不优 → 智能路由
- 分润核算困难 → 自动计算

---

### 流程B：退货处理全流程

**适用场景**：客户申请退货到退款完成

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  资格校验   │ →  │  退款处理   │ →  │  物流追踪   │
│ (7天无理由)  │    │ (原路返回)  │    │ (实时进度)  │
└─────────────┘    └─────────────┘    └─────────────┘
                                              ↓
                                        ┌─────────────┐
                                        │  质量检验   │
                                        │ (残次鉴定)  │
                                        └─────────────┘
                                              ↓
                                        ┌─────────────┐
                                        │  缺陷分类   │
                                        │ (责任认定)  │
                                        └─────────────┘
                                              ↓
                                        ┌─────────────┐
                                        │  库存再上架 │
                                        │ (合格品)    │
                                        └─────────────┘
```

**解决的问题**：
- 退货资格人工判断 → 自动校验
- 退款进度不透明 → 实时追踪
- 残次品堆积 → 及时分类处理
- 退货商品无法再次销售 → 自动再上架

---

### 流程C：对账全流程

**适用场景**：每日财务核对

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  账单核对   │ →  │  凭证生成   │ →  │  报表导出   │
│ (差异标注)  │    │ (财务对接)  │    │ (多维度)    │
└─────────────┘    └─────────────┘    └─────────────┘
```

**解决的问题**：
- 多平台账单人工比对 → 自动匹配
- 差异订单难追溯 → 标红差异项
- 财务凭证手工录入 → 自动生成
- 报表汇总耗时 → 一键导出

## 四、自定义场景配置

### 如何设计新流程

**Step 1：明确业务目标**
- 要解决什么问题？
- 涉及哪些角色（客户/客服/财务/仓库）？
- 关键节点有哪些？

**Step 2：选择技能组件**

| 目标 | 推荐技能 |
|------|----------|
| 识别用户 | oms-one-id-merge |
| 检查库存 | oms-inventory-realtime |
| 计算优惠 | oms-promotion-engine |
| 分配订单 | oms-order-routing |
| 处理退货 | oms-returns-crosschannel |
| 物流追踪 | oms-returns-logistics |
| 分润结算 | oms-profit-sharing |
| 财务核对 | oms-reconciliation |

**Step 3：配置数据流转**

```
输入数据 → 步骤1(技能A) → 输出数据 → 步骤2(技能B) → ...
```

**Step 4：设定业务规则**
- 超卖阈值是多少？
- 优惠叠加优先级？
- 分润比例？

---

### 示例：会员日营销活动

**目标**：会员日当天，提升老客户复购

**流程设计**：

```
1. 识别会员 → 2. 查询等级 → 3. 发放专属券 → 4. 库存锁定 → 5. 路由最优门店自提
```

**配置说明**：

| 步骤 | 技能 | 输入 | 输出 |
|------|------|------|------|
| 1 | 用户识别 | 手机号/会员ID | 会员基础信息 |
| 2 | 画像构建 | 会员ID | 等级、积分、历史购买 |
| 3 | 发优惠券 | 会员等级 | 专属折扣券 |
| 4 | 库存检查 | SKU+数量 | 是否可售 |
| 5 | 路由计算 | 客户地址 | 最优门店 |

**营销效果**：
- 会员专属感 → 参与率提升
- 门店自提 → 到店连带消费
- 库存锁定 → 避免超卖投诉

---

### 示例：预售商品尾款支付

**目标**：预售商品先预售后付尾款，保障库存

**流程设计**：

```
1. 预售下单 → 2. 预扣库存 → 3. 尾款支付 → 4. 正式占库存 → 5. 发货
```

**配置说明**：

| 步骤 | 技能 | 关键逻辑 |
|------|------|----------|
| 1 | 订单捕获 | 标记为预售订单 |
| 2 | 库存预扣 | 预留库存 |
| 3 | 支付处理 | 尾款到账通知 |
| 4 | 库存确认 | 预售转正 |
| 5 | 订单路由 | 发货 |

## 五、典型行业方案

### 服装行业

**痛点**：SKU多、库存分散、促销复杂

**推荐流程**：
```
订单捕获 → 颜色尺码校验 → 库存查询(多仓) → 促销计算(多券叠加) → 路由(最近门店) → 佣金计算
```

**亮点**：
- 多规格库存同步
- 多优惠券叠加计算
- 门店自提+连带销售

---

### 3C数码

**痛点**：高价商品防黄牛、售后保外维修

**推荐流程**：
```
订单捕获 → 黄牛检测 → 实名认证 → 库存检查 → 促销计算 → 路由 → 延保推荐 → 佣金计算
```

**亮点**：
- 黄牛订单拦截
- 延保增值服务推荐
- 平台服务费精确计算

---

### 生鲜食品

**痛点**：保鲜期短、损耗高、时效要求高

**推荐流程**：
```
订单捕获 → 库存查询(效期管理) → 冷藏包装校验 → 路由(最近冷链仓) → 优先发货 → 佣金计算
```

**亮点**：
- 效期优先匹配
- 冷链配送最优
- 损耗核算

---

### 美妆护肤

**痛点**：套装组合、赠品管理、过敏售后

**推荐流程**：
```
订单捕获 → 套装拆解 → 库存查询 → 赠品校验 → 促销计算(买赠) → 路由 → 售后(过敏保障) → 佣金
```

**亮点**：
- 套装拆分库存
- 赠品防超领
- 过敏无忧退

## 七、常见问题

**Q1：现有系统如何对接？**
> 通过API调用技能服务，输入JSON格式数据，获取结构化输出

**Q2：业务规则如何配置？**
> 每个技能的输入参数支持自定义，如佣金比例、优惠门槛等

**Q3：可以只启用部分流程吗？**
> 可以，按需选择单个技能或组合使用

**Q4：促销规则经常变怎么办？**
> 促销引擎支持参数化配置，运营可随时调整
