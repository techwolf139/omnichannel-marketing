# 跨平台引流归因模型

本文档定义 OMS 系统在"看小红书/抖音 → 天猫/京东/小程序下单"场景下的归因模型，解决多触点、多渠道的业绩归属问题。

## 1. 归因问题定义

### 1.1 典型场景

```
场景：顾客旅程

第1天：小红书看到某服装品牌笔记（未互动）
第3天：抖音刷到该品牌直播（观看但未下单）
第5天：收到企业微信推送的优惠券
第7天：在天猫下单购买
第15天：收到货后满意，门店试穿其他款式
第20天：在品牌小程序再次购买
```

**问题**：这笔天猫订单的"功劳"应该归谁？
- 小红书种草？
- 抖音直播？
- 企业微信推送？
- 或者是门店试穿？

### 1.2 归因难点

| 难点 | 说明 |
|------|------|
| 跨平台追踪 | 小红书/抖音/天猫 分属不同平台，OpenID不互通 |
| 时间衰减 | 越久远的触点贡献度越难衡量 |
| 辅助转化 | 部分触点只起辅助作用，不直接导致购买 |
| 沉默用户 | 用户可能多次触达后才购买 |

## 2. 归因模型

### 2.1 模型类型

| 模型 | 说明 | 适用场景 |
|------|------|----------|
| 首次触点归因 | 100%功劳归第一个触点 | 强调拉新效果 |
| 最终触点归因 | 100%功劳归最后一个触点 | 强调收割效果 |
| 最终非直接触点归因 | 功劳归最后一次引流触点（非直接购买渠道） | 区分引流与收割 |
| 线性归因 | 平均分配给所有触点 | 强调参与感 |
| 时间衰减归因 | 越接近购买的触点权重越高 | 强调时机 |
| 位置归因 | 首尾触点各占40%，中间触点均分20% | 平衡首尾 |
| 自定义归因 | 根据业务规则设置权重 | 按需配置 |

### 2.2 推荐模型

```
推荐：位置归因 + 时间衰减组合

首触点权重：40%（发现价值）
尾触点权重：40%（收割价值）
中间触点权重：20%（按时间衰减分配）
```

## 3. 触点类型定义

### 3.1 触点分类

| 触点类型 | 代码 | 说明 | 默认权重系数 |
|----------|------|------|--------------|
| 小红书种草 | XHS_CONTENT | 笔记曝光/互动 | 0.15 |
| 抖音/快手内容 | KS_CONTENT | 短视频/直播曝光 | 0.15 |
| 广告投放 | AD_IMPRESSION | 信息流广告曝光 | 0.20 |
| 广告点击 | AD_CLICK | 广告点击 | 0.25 |
| 企业微信 | WECHAT_WORK | 导购私聊/群发 | 0.20 |
| 门店体验 | STORE_VISIT | 线下门店试穿/咨询 | 0.25 |
| 品牌小程序 | MINI_PROGRAM | 品牌自有平台访问 | 0.15 |
| 平台搜索 | PLATFORM_SEARCH | 在购买平台搜索 | 0.10 |

### 3.2 触点归因权重表

```
场景：某顾客旅程

触点序列:
  1. 小红书笔记曝光      (XHS_CONTENT)
  2. 抖音直播观看        (KS_CONTENT)
  3. 企业微信推送优惠券  (WECHAT_WORK)
  4. 天猫下单购买        (PLATFORM_SEARCH)

权重计算（时间衰减 + 位置）:
  触点1: 0.30 × 0.4 = 0.12
  触点2: 0.25 × 0.3 = 0.075
  触点3: 0.25 × 0.3 = 0.075
  触点4: 购买转化 = 1.00

最终归因比例:
  小红书: 12%
  抖音: 7.5%
  企业微信: 7.5%
  天猫搜索: 73%
```

## 4. 技术实现

### 4.1 数据采集

```javascript
// 触点事件采集
TouchPointEvent {
  user_id: string,           // One-ID
  touch_type: string,        // 触点类型
  platform: string,           // 平台
  content_id: string,        // 内容ID（笔记/视频/商品）
  exposure_time: datetime,    // 曝光时间
  click_time: datetime,       // 点击时间
  utm_source: string,        // UTM来源
  utm_medium: string,        # UTM媒介
  utm_campaign: string,       // UTM活动
  device_id: string,         // 设备ID（脱敏）
  ip: string,                // IP地址
  user_agent: string         // 浏览器信息
}
```

### 4.2 归因计算流程

```
订单完成 → 追溯用户触点序列
              ↓
        筛选归因窗口（订单前30天内）
              ↓
        按模型计算各触点权重
              ↓
        分润记录写入 profit_sharing
              ↓
        生成业绩归属报表
```

### 4.3 归因窗口

| 窗口类型 | 时间范围 | 适用场景 |
|----------|----------|----------|
| 点击归因 | 点击广告后7天 | 广告投放 |
| 曝光归因 | 广告曝光后30天 | 品牌广告 |
| 跨渠道归因 | 任意触点后30天 | 全链路分析 |

## 5. 业绩归属应用

### 5.1 分润场景

| 场景 | 归属方向 | 分润规则 |
|------|----------|----------|
| 小红书种草→天猫下单 | 小红书内容团队 | 按归因比例计算 |
| 抖音直播→小程序下单 | 抖音运营团队 | 按归因比例计算 |
| 导购引流→门店试穿→下单 | 导购+门店 | 按贡献度分配 |
| 企业微信推送→下单 | 企业微信团队 | 按归因比例计算 |

### 5.2 结算周期

```
归因结算:
  - 订单完成后 → 实时计算归因
  - 退货订单 → 归因结果冲销
  - 每月1日 → 上月归因数据锁定
  - 每月5日 → 生成归因报表
```

## 6. 归因挑战与处理

### 6.1 挑战

| 挑战 | 处理方式 |
|------|----------|
| 跨平台OpenID不互通 | 手机号关联 + 行为相似度 |
| 用户清除Cookie | Device ID + 概率模型 |
| 线下触点无法追踪 | 导购手动录入 + 门店WiFi探针 |
| 数据延迟 | 异步归因 + 定期校正 |

### 6.2 归因校准

```
归因结果校正:
  1. 样本验证：抽样回访用户确认旅程
  2. 趋势校验：对比归因前后数据波动
  3. A/B测试：对比不同归因模型效果
  4. 机器学习：训练模型预测归因准确率
```

## 7. 归因报表

### 7.1 核心指标

| 指标 | 说明 |
|------|------|
| 归因订单量 | 各触点带来的订单数 |
| 归因GMV | 各触点带来的销售额 |
| 归因转化率 | 触点UV → 订单转化率 |
| 触点ROI | 触点投入成本 → 归因GMV |
| 归因占比 | 各触点占总归因GMV比例 |
