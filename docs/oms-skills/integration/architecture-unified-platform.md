# 跨平台统一系统架构设计

本文档定义跨越不同销售平台（天猫/京东/拼多多）和营销推广平台（小红书/抖音/微信）的统一系统架构。

## 1. 架构愿景

**目标**：构建一个"以消费者为中心"的数据中台，实现：
- 订单统一管理
- 流量统一运营
- 推荐统一服务
- 身份统一识别

**核心价值**：
```
跨平台数据孤岛 ──→ 统一数据中台
      │
      ├── 消费者：跨平台一致体验
      ├── 品牌商：全链路数据洞察
      ├── 门店：数字化运营能力
      └── 平台：高效协同
```

## 5. API 设计

### 5.1 统一身份 API

| API | 方法 | 说明 |
|-----|------|------|
| `/identity/resolve` | POST | 通过任意标识查询 One-ID |
| `/identity/merge` | POST | 手动合并身份 |
| `/identity/split` | POST | 拆分误合并身份 |
| `/identity/profile` | GET | 获取用户画像 |

### 5.2 统一订单 API

| API | 方法 | 说明 |
|-----|------|------|
| `/orders` | GET | 查询统一订单列表 |
| `/orders/{id}` | GET | 查询订单详情 |
| `/orders/sync` | POST | 触发订单同步 |
| `/orders/{id}/route` | POST | 手动触发路由 |

### 5.3 统一流量 API

| API | 方法 | 说明 |
|-----|------|------|
| `/events/track` | POST | 上报行为事件 |
| `/crowds` | GET | 查询人群包列表 |
| `/crowds/{id}/export` | POST | 导出人群包 |
| `/attribution/calculate` | POST | 计算归因 |

### 5.4 统一推荐 API

| API | 方法 | 说明 |
|-----|------|------|
| `/recommend/product` | GET | 商品推荐 |
| `/recommend/content` | GET | 内容推荐 |
| `/recommend/search` | GET | 搜索排序 |
| `/recommend/feedback` | POST | 推荐反馈 |

## 6. 技术选型

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 身份存储 | Neo4j + Redis | 图数据库存储关系，Redis缓存 |
| 订单存储 | MySQL + Elasticsearch | 事务处理 + 搜索 |
| 行为数据湖 | Kafka + Iceberg | 实时采集 + 湖存储 |
| 流处理 | Apache Flink | 实时流处理 |
| 特征存储 | Feast + Redis | 特征工程 |
| 推荐服务 | TensorFlow Serving | 模型服务 |
| API网关 | Kong / APISIX | 统一入口 |
| 容器编排 | Kubernetes | 弹性伸缩 |

## 8. 容灾与高可用

| 措施 | 说明 |
|------|------|
| 多活部署 | 异地多活，主备切换 < 30s |
| 数据备份 | 实时同步 + 定期全量 |
| 限流降级 | 平台 API 限流时的降级策略 |
| 熔断机制 | 防止故障扩散 |
| 监控告警 | 全链路监控 + 异常告警 |
