# 智能订单路由与拆合单 - 设计文档

## 1. 设计目标

构建订单路由决策大脑，根据库存分布、地址位置、配送成本、时效要求等多维度因素，自动计算并执行最优履约路径。

## 2. 核心概念

### 2.1 履约网络结构

```
FulfillmentNetwork {
  warehouses: [
    {id, name, type: rdc|cdc|fdc, location, inventory}
  ],
  stores: [
    {id, name, location, inventory, capability: pickup|deliver}
  ],
  delivery_partners: [
    {id, name, type: sf|jt|mt|ele, coverage, cost_model}
  ]
}
```

### 2.2 路由决策算法

**成本-时效加权模型**:

```
Score = α × Cost + β × Time + γ × Distance

参数说明:
- α: 成本权重（默认0.4）
- β: 时效权重（默认0.4）
- γ: 距离权重（默认0.2）
- 根据业务场景动态调整
```

**库存可用性检测**:

| 库存状态 | 说明 | 路由决策 |
|----------|------|----------|
| AVAILABLE | 立即可用 | 参与路由计算 |
| RESERVED | 已预占 | 等待释放或路由其他 |
| IN_TRANSIT | 在途 | 等待入库或路由其他 |
| RINGFENCED | 隔离库存 | 不参与公域路由 |

### 2.3 拆合单决策

**拆分条件**:
- 商品跨多个仓库/门店
- 拆分后每单都满足最低包邮门槛
- 拆分后时效明显改善

**合单条件**:
- 同一顾客24小时内的多笔订单
- 订单均未进入履约流程
- 合单后不超重/超体积

## 3. 核心能力

### 3.1 路由计算

- 实时计算全网最优履约路径
- 支持指定仓库/门店的强制路由
- 支持排除特定配送方式的路由

### 3.2 拆合单执行

- 自动生成调拨单（仓间调拨）
- 自动生成合单打包任务
- 记录拆合单原因供审计

## 4. 能力边界

### 4.1 支持的操作

- `calculate_route`: 计算最优路由
- `allocate_order`: 执行订单分配
- `split_order`: 拆分订单
- `merge_orders`: 合并订单
- `get_routing_detail`: 查询履约路径

### 4.2 限制说明

- 路由计算超时：3秒内必须返回
- 拆单最多支持拆分为5个子单
- 已发货订单不支持合单

## 5. 错误处理

| 错误码 | 说明 | 处理方式 |
|--------|------|----------|
| NO_INVENTORY | 全网无货 | 订单挂起，通知采购 |
| ROUTING_TIMEOUT | 路由计算超时 | 降级为成本优先策略 |
| ALLOCATION_FAILED | 分配失败 | 推送人工处理队列 |
| CANNOT_SPLIT | 不满足拆分条件 | 返回不可拆原因 |
